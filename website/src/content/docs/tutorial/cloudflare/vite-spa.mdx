You have a Worker, a Durable Object, and a chat `Room` that
broadcasts WebSocket messages — enough backend for an interesting
app, but no UI. In this part you'll add a Vite single-page app
that builds and deploys from the same `alchemy.run.ts` as your
Worker, with no separate build pipeline or Wrangler config.

`Cloudflare.Vite` invokes Vite programmatically (via
`createBuilder`) with the `@cloudflare/vite-plugin` already wired
in. It builds your client assets, ships them to Cloudflare, and
serves them through a Worker — so your frontend and backend share
one URL surface and one deploy.

## What `Cloudflare.Vite` actually requires

The bare minimum is **an `index.html` with a script tag**. That's
it. No `vite.config.ts`, no `package.json` `build` script, no
`main` entry. Alchemy supplies the Vite config and the
`@cloudflare/vite-plugin` itself.

A typical SPA layout — whether at the project root or in a
subfolder — looks like:

```
.                       # or web/, app/, frontend/, client/ — your call
├── index.html          # entry HTML, references the client bundle
└── src/
    └── main.ts         # client entry imported by index.html
```

`index.html` just needs to load your entry module:

```html
<!doctype html>
<html lang="en">
  <head><meta charset="UTF-8" /><title>My App</title></head>
  <body>
    <div id="app"></div>
    <script type="module" src="/src/main.ts"></script>
  </body>
</html>
```

That's the contract. Use any framework Vite supports (vanilla TS,
React, Vue, Solid, Svelte, …) — bring whatever `src/` layout the
framework wants.

:::tip[Use root, subfolder, whatever]
`rootDir` defaults to `process.cwd()`. Drop `index.html` next to
your `alchemy.run.ts` and you're done — no subfolder required, no
duplicate `package.json`. If you prefer separation (e.g. you want
the SPA's deps isolated from the Worker's), put it in `web/` and
pass `rootDir: "./web"`. Either way works; pick what fits the
project.
:::

## Add `Cloudflare.Vite` to the Stack

Configure asset handling for SPA fallback (every unmatched route
serves `index.html`) and enable `nodejs_compat` (the Cloudflare
Vite plugin's runtime shim imports from `node:*`):

```diff lang="typescript"
// alchemy.run.ts
import * as Alchemy from "alchemy";
import * as Cloudflare from "alchemy/Cloudflare";
import * as Effect from "effect/Effect";
import Worker from "./src/worker.ts";

+const Web = Cloudflare.Vite("Web", {
+  // rootDir: "./web",   // omit to use the project root
+  compatibility: { flags: ["nodejs_compat"] },
+  assets: {
+    config: {
+      htmlHandling: "auto-trailing-slash",
+      notFoundHandling: "single-page-application",
+    },
+  },
+});

export default Alchemy.Stack(
  "MyApp",
  {
    providers: Cloudflare.providers(),
    state: Cloudflare.state(),
  },
  Effect.gen(function* () {
    const worker = yield* Worker;
+    const web = yield* Web;
    return {
      url: worker.url,
+      webUrl: web.url,
    };
  }),
);
```

`notFoundHandling: "single-page-application"` is what makes
client-side routing work — a deep link like `/about` returns
`index.html` instead of a 404, and your router handles it on the
client.

## What NOT to do

A few things that look helpful but break the build:

- **Don't add `build.rollupOptions.input` to a custom
  `vite.config.ts`.** Alchemy uses Vite's multi-environment build
  with both a `client` and an `ssr` environment. A top-level
  `rollupOptions.input` leaks into the `ssr` environment and
  collides with the Cloudflare plugin's server entry. If you
  don't have an SSR framework, skip `vite.config.ts` entirely.
- **Don't forget `nodejs_compat`.** Without it, the upload fails
  with `Cannot find module 'node:module'`.

## Deploy

```sh
bun alchemy deploy
```

Alchemy runs Vite on `rootDir`, uploads the assets, creates a
Worker that serves them, and prints the new `webUrl` stack output:

```
{
  url: "https://myapp-worker-dev-you.workers.dev",
  webUrl: "https://myapp-web-dev-you.workers.dev",
}
```

## Verify

Hit it with `curl`:

```sh
curl -s https://myapp-web-dev-you.workers.dev | head -5
# <!doctype html>
# <html lang="en">
#   <head>
#     <meta charset="UTF-8" />
#     ...
```

Or add an integration test alongside the existing ones:

```diff lang="typescript"
// test/integ.test.ts
import { beforeAll, deploy, expect, test } from "alchemy/Test/Bun";
import * as Effect from "effect/Effect";
import * as HttpClient from "effect/unstable/http/HttpClient";
import Stack from "../alchemy.run.ts";

const stack = beforeAll(deploy(Stack));

+test(
+  "Web SPA serves index.html",
+  Effect.gen(function* () {
+    const { webUrl } = yield* stack;
+
+    const response = yield* HttpClient.get(webUrl);
+    expect(response.status).toBe(200);
+    expect(yield* response.text).toContain("<!doctype html>");
+  }),
+);
```

## Local dev

`alchemy dev` (from [Tutorial Part 4](/tutorial/part-4)) runs Vite,
your Worker, and any Durable Objects / Containers / Workflows you
added — all under one watcher. Edit a `.ts` in `src/` and the
browser hot-reloads against the live Cloudflare resources you
deployed earlier.

```sh
alchemy dev
```

## Use a different framework

`Cloudflare.Vite` works with any Vite-based framework. The setup
above is the SPA case (no SSR). For SSR frameworks, drop the
`assets.config` block and let the framework's Vite plugin own the
entry. Common choices:

- **TanStack Start** — full-stack React with file-based routing and
  server functions. See [examples/cloudflare-tanstack](https://github.com/alchemy-run/alchemy-effect/tree/main/examples/cloudflare-tanstack).
- **SolidStart** — SSR Solid with file-based routing. See
  [examples/cloudflare-solidstart](https://github.com/alchemy-run/alchemy-effect/tree/main/examples/cloudflare-solidstart).
- **SolidJS SSR (manual)** — when you want full control over the
  SSR pipeline rather than SolidStart's conventions. See
  [examples/cloudflare-solidjs-ssr](https://github.com/alchemy-run/alchemy-effect/tree/main/examples/cloudflare-solidjs-ssr).
- **Vue 3 SPA** — Vite's default Vue template, with the same
  `notFoundHandling: "single-page-application"` config as above.
  See [examples/cloudflare-vue](https://github.com/alchemy-run/alchemy-effect/tree/main/examples/cloudflare-vue).
- **Plain static site** — a single `index.html` (no framework)
  ships through `Cloudflare.Vite("Website")` with no extra config.
  See [examples/cloudflare-static-site](https://github.com/alchemy-run/alchemy-effect/tree/main/examples/cloudflare-static-site).

For deeper coverage of framework-specific patterns (asset configs,
SSR vs SPA tradeoffs, custom build steps), see the
[Frontend frameworks guide](/guides/frontends).
